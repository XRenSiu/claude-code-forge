## P1 — 质量

### P1-1: `unit/premium_access.test.ts` 在文件末尾再次 `import` `CancelSubscriptionUseCase`，与文件顶部 import block 分裂

- **证据**：worker 产出 §4.3 行 864：
  > ```typescript
  > // Re-import for the monotonic test above (CancelSubscriptionUseCase is the cancel
  > // op; checkPremiumAccess / transitionAtPeriodEnd are the access / boundary ops).
  > import { CancelSubscriptionUseCase } from '../../../src/billing/cancel_subscription_use_case';
  > ```
  此 import 出现在最后一个 `test()` 体之后，文件顶部 (worker §4.3 行 684-690) 已经从同一模块 import 了 `checkPremiumAccess` / `transitionAtPeriodEnd` / `Subscription` 但漏 import `CancelSubscriptionUseCase`，于是补在末尾。
- **依据**：FLOW.md 行 45 字面要求 "unit example + PBT (Hypothesis/fast-check)"。一份产出的 PBT 文件 (`test_status_transition_is_monotonic_active_to_cancelled_active_to_expired` worker §4.3 行 812-860) 在文件中 line 837 已经直接调用了 `CancelSubscriptionUseCase`，意味着这个 PBT 在文件被 TypeScript 编译/vitest 装载时无法解析符号——文件本身不可运行。
- 这条算 P1（FLOW.md 要 unit batch 产出的 PBT，worker 确实 emit 了一条 monotonic PBT，但 emit 出来的文件因 import 错位而不可直接跑），对应 P1 例 "PBT 识别的属性过于浅层"——这里属性本身不浅，但 emit 的载体不可用，等价于 PBT 没真正落地。

### P1-2: 4-A existence.sh 漏盖 `function: CancelSubscriptionUseCase` 的"exported"语义校验，rg pattern 在多种合法 TS 导出形态下会假阳性

- **证据**：worker 产出 §4.1 行 367-368：
  > ```bash
  > check "function: CancelSubscriptionUseCase exported" \
  >   rg -q "export\s+(class|function|const)\s+CancelSubscriptionUseCase" src/
  > ```
  对应 done_when.yaml 行 8 `function: CancelSubscriptionUseCase`。该正则只能匹配 `export class|function|const CancelSubscriptionUseCase`，但 TS 中还有两种合法导出形态会被假阴性挡掉：
  1. 命名重导出 `export { CancelSubscriptionUseCase }`（barrel 文件常见）
  2. `export default class CancelSubscriptionUseCase`
- **依据**：FLOW.md 行 44 字面要求 `4-A existence ripgrep/tree-sitter 检查`，且 existence 一栏在 done_when.yaml 是契约性指代——worker §4.1 emit 的 rg 模式不能完整匹配 done_when.yaml 同一行所指的 `function:` 概念。
- 这条算 P1，对应 P1 例 "existence 检查只覆盖部分名词指代"——具名导出和 default 导出两条合法路径未被 rg pattern 覆盖。
