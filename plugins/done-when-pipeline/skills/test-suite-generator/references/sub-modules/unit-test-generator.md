# Sub-step 4-B: Unit test generator

**Goal:** turn each `behavior.unit_tests` entry in `done_when.yaml` into runnable test code. Test single functions / classes, no I/O.

**Output:** one test file per source module under test (`tests/<feature>/unit/test_<module>.<ext>`).

---

## Per-test recipe

For every entry in `behavior.unit_tests.example_based[]` and `behavior.unit_tests.property_based[]`:

1. Find the source REQ(s) from `based_on:` in the YAML.
2. Read those REQs from `spec.md`.
3. Apply the template for that REQ's EARS type (see `../ears-to-test-matrix.md`).
4. For PBT entries: read `property_type:`, apply the matching pattern from `../pbt-property-types.md`.
5. Emit the test, with the file header below.

---

## File header (mandatory)

Every generated unit test file starts with:

```python
# Generated by test-suite-generator/0.1.0 — do not edit by hand.
# To regenerate: re-run /test-suite-generator <path/to/done_when.yaml>
#
# Source spec: <relative path to spec.md>
# Tests in this file: <comma-separated list>
# REQ coverage: <comma-separated REQ-IDs>
```

(Adjust comment syntax for the language: `//` for TS, `//` for Swift, `//` for Kotlin.)

## Example-based template (per WHEN-THEN REQ)

For an Event-driven REQ like:
> `WHEN the user clicks "cancel" AND status is active, THE system SHALL set status to cancelled_active.`

```python
def test_cancel_active_sets_status_to_cancelled_active():
    """Verifies REQ-001: WHEN user clicks cancel on active sub → status becomes cancelled_active."""
    sub = make_subscription(status='active', end_date=date(2025, 1, 31))
    cancel(sub)
    assert sub.status == 'cancelled_active'
    assert sub.cancelled_at is not None
```

Rules:
- One test = one observable consequence of one REQ. Do not stuff three assertions about three REQs into one test.
- Use a factory (`make_subscription`) — do not hand-construct objects with 10 fields inline.
- Assert on observable state, not on internal calls (no `mock.assert_called_with` if you can help it).

## PBT template (per property type)

Read `../pbt-property-types.md` for full patterns. Quick reference:

| `property_type:` | Python (Hypothesis) skeleton |
|---|---|
| `invariant` | `@given(operations_strategy()) def test_X_invariant(ops): ...` |
| `idempotent` | `@given(input_strategy()) def test_op_idempotent(x): op(x); s1=snap(); op(x); s2=snap(); assert s1==s2` |
| `reversible` | `@given(input_strategy()) def test_inverse(x): before=snap(x); do(x); undo(x); assert snap(x)==before` |
| `boundary` | `@given(st.integers(min_value=B-2, max_value=B+2)) def test_boundary(n): assert behavior(n) == expected(n)` |
| `monotonic` | `@given(sorted_lists()) def test_monotonic(xs): prev=None; for x in xs: y=f(x); assert prev is None or y>=prev; prev=y` |
| `state_machine` | use `RuleBasedStateMachine`; see full pattern in pbt-property-types.md |

Every PBT must include a docstring naming the property type and the source REQ:

```python
@given(active_subscriptions())
@settings(max_examples=500, deadline=None)
def test_cancel_is_idempotent(sub):
    """Property: idempotent (REQ-001). cancel(cancel(x)) ≡ cancel(x)."""
    cancel(sub); s1 = sub.snapshot()
    cancel(sub); s2 = sub.snapshot()
    assert s1 == s2
```

`max_examples=500` corresponds to `thresholds.pbt_runs_per_property` in the YAML; honor that number.

## Per-language idioms

### Python (pytest + Hypothesis)

```python
import pytest
from hypothesis import given, strategies as st, settings, example
from datetime import date, datetime, timedelta

from src.billing.cancel import cancel
from src.billing.subscription import Subscription

def make_active_subscription(end_date=None):
    return Subscription(
        status='active',
        start_date=date(2025, 1, 1),
        end_date=end_date or date(2025, 1, 31),
    )

@st.composite
def active_subscriptions(draw):
    end = draw(st.dates(min_value=date.today(), max_value=date(2030, 1, 1)))
    return make_active_subscription(end_date=end)
```

### TypeScript (vitest + fast-check)

```typescript
import { test, expect, describe } from 'vitest';
import * as fc from 'fast-check';
import { cancel, type Subscription } from '../../src/billing/cancel';

const activeSub = (): fc.Arbitrary<Subscription> => fc.record({
  status: fc.constant('active'),
  endDate: fc.date({ min: new Date(), max: new Date('2030-01-01') }),
});

test('cancel is idempotent (REQ-001)', () => {
  fc.assert(fc.property(activeSub(), (sub) => {
    cancel(sub);
    const s1 = JSON.stringify(sub);
    cancel(sub);
    const s2 = JSON.stringify(sub);
    return s1 === s2;
  }), { numRuns: 500 });
});
```

### Swift (XCTest, example-only)

```swift
import XCTest
@testable import Billing

final class CancelSubscriptionTests: XCTestCase {
```

## When to NOT generate a unit test

- The REQ's `based_on:` only references a Ubiquitous REQ that is genuinely a cross-cutting policy (e.g. "all stored passwords SHALL be encrypted") — that's an integration concern, not a unit one. Move it to 4-C.
- The REQ refers exclusively to UI presentation (e.g. "shows a dialog") — that's e2e or component test, not unit. Move to 4-D.
- The `based_on:` REQ does not exist in `spec.md` — validator should have caught this; bail out.

---

## Quality checklist before writing to disk

- [ ] Every test has a docstring/comment with the REQ ID it verifies
- [ ] Every PBT has `property_type` named in the docstring
- [ ] Every PBT has `max_examples` / `numRuns` matching `pbt_runs_per_property` threshold
- [ ] No `assert True`, no `assert 1 == 1`, no skipped tests
- [ ] No `@pytest.skip` without a TODO comment explaining why and pointing at an issue
- [ ] No `time.sleep()` (unit tests must not be timing-dependent)
- [ ] No I/O (filesystem, network, DB) — those belong in 4-C
