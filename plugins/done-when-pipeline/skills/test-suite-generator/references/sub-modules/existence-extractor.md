# Sub-step 4-A: Existence extractor

**Goal:** turn each `existence:` entry in `done_when.yaml` into one shell-runnable check. Fast, **fail-fast** (no test framework needed).

**Output:** `existence.sh` (Unix-friendly stacks) or `existence.py` (Python-only or Windows-first projects).

---

## Hard rule: fail-fast (set -e), not count-all

Per `done-when-pipeline.md` §6.2, the script MUST stop on the **first** failing check. Its job is to give the ratchet evaluator a quick "missing X" signal — not a tally of every missing thing.

**Required idiom** — the failure branch ends in `exit 1` (or `return 1` from a helper that the caller then `||`-chains to `exit 1`). The success branch returns 0 and falls through.

**Forbidden idiom** — wrapping the assertion in `if … then PASS=… else FAIL=… fi`. The `if` context disables `errexit`, so failures are quietly counted instead of aborting. Same for any final "summary" block that runs after all checks have been evaluated.

---

## The template

```bash
#!/usr/bin/env bash
# Generated by test-suite-generator/0.1.0
# Source: <relative path to done_when.yaml>
# Coverage: all <N> existence claims in done_when.yaml at time of generation.
# Run from project root. Exits nonzero on the FIRST failure (set -e semantics).

set -euo pipefail

# Fail-fast helper. Echoes ✓ on success, ✗ + exit 1 on first failure.
# Do NOT wrap the body in `if … fi` — that disables errexit and turns this
# into a count-all-then-exit script, which violates done-when-pipeline §6.2.
check() {
  local label="$1"; shift
  "$@" >/dev/null 2>&1 || { echo "✗ FAIL: $label" >&2; exit 1; }
  echo "✓ $label"
}

# === existence checks ===

# file: src/billing/cancel_subscription_use_case.ts  (based_on: REQ-001)
check "file: src/billing/cancel_subscription_use_case.ts" \
  test -f src/billing/cancel_subscription_use_case.ts

# function: CancelSubscriptionUseCase  (based_on: REQ-001)
check "function: CancelSubscriptionUseCase exported" \
  rg -q "export\s+(class|function|const)\s+CancelSubscriptionUseCase" src/

# route: POST /api/subscription/cancel  (based_on: REQ-001, REQ-004)
check "route: POST /api/subscription/cancel registered" \
  rg -q "['\"]/api/subscription/cancel['\"]" src/

# db_field: subscription.cancelled_at  (based_on: REQ-001)
check "db_field: subscription.cancelled_at" \
  rg -q "cancelled_at" src/db/

# frontend_component: CancelSubscriptionButton  (based_on: REQ-001)
check "frontend_component: CancelSubscriptionButton" \
  rg -q "CancelSubscriptionButton" src/components/

# === summary ===
# Reached only if every check above passed (fail-fast).
echo "✓ All existence checks passed"
```

### Forbidden pattern (do NOT generate this)

```bash
# WRONG — `if` disables `set -e`; failed checks increment FAIL but the script keeps going.
check() {
  local label="$1"; shift
  if "$@" >/dev/null 2>&1; then
    PASS=$((PASS+1))
    echo "✓ $label"
  else
    FAIL=$((FAIL+1))
    echo "✗ $label" >&2
  fi
}

# … all checks …

# WRONG — `Passed: X · Failed: Y` tally followed by `if [[ FAIL > 0 ]]; then exit 1`.
# This is count-all-then-exit. Violates §6.2.
```

If a user explicitly requests a count-all diagnostic mode (e.g. for CI summary), emit it as a **separate** `existence-diag.sh` with a clearly different name and documented intent — never overwrite the fail-fast default.

## Per-kind check recipes

Schema v1 (Appendix C of `done-when-pipeline.md`) defines **five** existence kinds. This table covers all of them:

| Kind | shell check | when to upgrade to AST |
|---|---|---|
| `file:` | `test -f <path>` | never |
| `function:` | `rg -q "export\\s+(class\\|function\\|const)\\s+<Name>" src/` | if name is ambiguous (matches many unrelated things) → upgrade to `ts-morph` (TS) / Python `ast` (Python) |
| `route:` | `rg -q "['\"]/<path>['\"]" src/` *plus* method-aware regex if router is structured | upgrade if framework uses non-string routing (decorators, magic) |
| `db_field:` | `rg -q "<column>" src/db/ src/migrations/` | upgrade if project uses a schema DSL (Prisma, SQLAlchemy declarative) — use the DSL's introspection instead |
| `frontend_component:` | `rg -q "<Name>" src/components/` | rarely needed |

If a contract uses any other kind (e.g. `env_var:` / `cli_command:` — both showed up in earlier drafts of this skill), it is non-v1 and should have been rejected at the validator step. Bail out and tell the user to regenerate the contract via `/acceptance-spec`.

**Rule of thumb:** ripgrep is fine for ~90% of cases. Reach for tree-sitter / language-native AST only when the simple grep produces false positives (the name is too generic and matches unrelated code) or false negatives (the symbol exists but is generated/macro-expanded).

## tree-sitter fallback for ambiguous names

If `rg` matches dozens of unrelated places for a function name like `cancel`, switch to:

```bash
tree-sitter query --captures \
  '(function_declaration name: (identifier) @fn-name (#eq? @fn-name "cancel"))' \
  src/billing/*.ts
```

Or in Python:

```python
import ast, sys
tree = ast.parse(open("src/billing/cancel.py").read())
assert any(
  isinstance(n, ast.FunctionDef) and n.name == "cancel"
  for n in ast.walk(tree)
), "function `cancel` not defined in cancel.py"
```

For TypeScript, `ts-morph` (`npm i -D ts-morph`) gives a clean object API:
```typescript
import { Project } from 'ts-morph';
const p = new Project({ tsConfigFilePath: 'tsconfig.json' });
const file = p.getSourceFileOrThrow('src/billing/cancel.ts');
if (!file.getFunction('CancelSubscriptionUseCase') && !file.getClass('CancelSubscriptionUseCase')) {
  console.error('✗ CancelSubscriptionUseCase not defined');
  process.exit(1);
}
```

## What this script does not do

- It does **not** check that the function does the right thing — that's 4-B/4-C.
- It does **not** check that the route returns the right response — that's 4-C.
- It does **not** check that the DB column has the right type — write a migration-level test in 4-C if you need that.

Existence is *fast-fail* before behavior. If existence fails, behavior tests are guaranteed to fail anyway, and existence tells you why with one grep instead of a stacktrace.

## Test-only endpoints / hooks MUST be in `existence:`

If the 4-C integration or 4-D e2e tests call any endpoint or hook that is *not part of the user-facing contract* — e.g. `/api/_test/seed_member`, `/api/_internal/surface_log`, an `_force_surface_outcomes` body field, or any other test side door — those endpoints MUST be in `done_when.yaml.existence:` as `route:` entries (or `function:` for non-HTTP hooks).

If `done_when.yaml` does NOT list them, the test suite implicitly demands a bigger contract than the contract makes explicit. That is spec drift: the implementer will ship the test-only routes (because integration tests would otherwise fail), but those routes are not tied back to any REQ. Step 5/6 reviewers cannot grep them; future maintainers cannot tell whether they are first-class or accidental.

**Correct workflow when the integration suite needs a test hook:**

1. While drafting 4-C/4-D, list every test-only endpoint/hook your tests will call.
2. Before writing those tests, check whether `done_when.yaml.existence:` already covers them as `route:` entries.
3. If not, **push back upstream**: tell the user "the contract is missing test-hook routes X, Y, Z — regenerate via `/acceptance-spec` to include them".
4. Acceptable alternative (faster but noisier): **augment `done_when.yaml` here** with the missing `route:` entries, and **explicitly note the augmentation** in the manifest you emit at the end of the skill. Never silently demand undeclared routes.

This rule applies symmetrically to `function:` (private helpers the tests poke), `db_field:` (test-only columns), and `frontend_component:` (test harness components). If a test references it and the production app must ship it, it belongs in `existence:`.
