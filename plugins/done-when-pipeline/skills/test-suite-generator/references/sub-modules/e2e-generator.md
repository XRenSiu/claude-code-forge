# Sub-step 4-D: E2E test generator

**Goal:** turn each `behavior.e2e_tests` entry into a browser/device automation script that exercises the real, deployed application from a user's perspective.

**Output:** `tests/<feature>/e2e/<journey>.spec.ts` (or equivalent).

---

## Rules of E2E generation

1. **Short list, only.** E2E tests are slow (10-60s each), flaky (timing, DOM races), and expensive (CI minutes). If `done_when.yaml.behavior.e2e_tests` lists more than ~5 entries, push back to the user before generating.
2. **One journey per test.** Cover the single golden path. Variants (different inputs, edge cases) belong at the integration layer.
3. **`data-test=` (or `data-testid=`) selectors only.** Selecting by text breaks on i18n; selecting by CSS class breaks on refactor. The doc convention is `data-test="cancel-subscription"`.
4. **No fixed sleeps.** Use the framework's wait primitives (`page.waitForSelector`, `expect(...).toBeVisible()`). `await sleep(3000)` is a smell — replace with a wait on an observable condition.
5. **Per-test data isolation.** Each test creates its own user / subscription / data — never share state between tests. Use the same fixture pattern as integration.

## Playwright (default for web)

```typescript
// tests/<feature>/e2e/cancel-subscription.spec.ts
// Generated by test-suite-generator/0.1.0
// Source REQ: REQ-001, REQ-003
import { test, expect } from '@playwright/test';
import { signInAsPaidUser, createSubscriptionForUser } from './fixtures';

test('user can cancel active subscription and sees expiry notice', async ({ page }) => {
  const user = await signInAsPaidUser(page);   // navigates, returns logged-in
  await createSubscriptionForUser(user);

  await page.goto('/account/subscription');
  await page.click('[data-test=cancel-subscription]');

  // confirmation modal
  await expect(page.locator('[data-test=confirm-cancel-modal]')).toBeVisible();
  await page.click('[data-test=confirm-cancel]');

  // success state
  await expect(page.locator('[data-test=cancel-success]')).toBeVisible();

  // expiry notice now appears on home (REQ-003)
  await page.goto('/');
  await expect(page.locator('[data-test=expiry-notice]')).toBeVisible();
  await expect(page.locator('[data-test=expiry-notice]')).toContainText(/expires on/i);
});
```

Configuration (`playwright.config.ts`):
```typescript
export default {
  testDir: './tests',
  timeout: 30_000,
  expect: { timeout: 5_000 },
  use: { baseURL: process.env.E2E_BASE_URL ?? 'http://localhost:3000' },
  reporter: [['list'], ['html', { open: 'never' }]],
};
```

## Cypress (only if project already uses it)

```javascript
// cypress/e2e/cancel-subscription.cy.js
describe('Cancel subscription flow (REQ-001, REQ-003)', () => {
  beforeEach(() => {
    cy.signInAsPaidUser();
    cy.createActiveSubscription();
  });

  it('user can cancel and sees expiry notice', () => {
    cy.visit('/account/subscription');
    cy.get('[data-test=cancel-subscription]').click();
    cy.get('[data-test=confirm-cancel]').click();
    cy.get('[data-test=cancel-success]').should('be.visible');
    cy.visit('/');
    cy.get('[data-test=expiry-notice]').should('contain.text', 'expires');
  });
});
```

## Mobile

### Appium (Android + iOS)

Use only if the project already has an Appium setup. Slow, finicky. Capabilities config (`wdio.conf.js` or equivalent) and a similar test structure to Playwright.

### Maestro (recommended for mobile)

Simpler than Appium for new projects. YAML-defined flows:

```yaml
# tests/<feature>/e2e/cancel-subscription.maestro.yaml
appId: com.example.app
---
- launchApp
- tapOn: "Sign In"
- inputText: "paid_user@test.com"
- tapOn: "Continue"
- tapOn: "Account"
- tapOn: "Cancel Subscription"
- tapOn: "Confirm"
- assertVisible: "Your subscription will expire on"
```

## API E2E

For backend-only systems where there is no UI:

| Tool | When to use |
|---|---|
| **k6** | load + functional; can be used for E2E with low VU count |
| **Newman** (Postman runner) | if the team already maintains Postman collections |
| **plain httpx/supertest scripts** | usually fine; promote integration-style tests to "E2E" by running them against the deployed staging env |

A plain Python script as a "deploy smoke test":

```python
# tests/<feature>/e2e/smoke_cancel.py
import os, sys, httpx
BASE = os.environ["E2E_BASE_URL"]
TOKEN = os.environ["E2E_TOKEN"]
r = httpx.post(f"{BASE}/api/subscription/cancel",
               headers={"Authorization": f"Bearer {TOKEN}"},
               json={"id": os.environ["E2E_SUB_ID"]},
               timeout=30.0)
assert r.status_code == 200, f"got {r.status_code}: {r.text}"
assert r.json()["status"] == "cancelled_active"
print("✓ smoke cancel passed")
```

Run in CI against staging post-deploy.

## Visual regression (special case)

If `done_when.yaml` mentions visual fidelity (the spec referenced UI consistency or matched a design), pair the e2e test with Playwright's screenshot diffing:

```typescript
test('cancel modal looks like the design', async ({ page }) => {
  await page.goto('/account/subscription');
  await page.click('[data-test=cancel-subscription]');
  await expect(page).toHaveScreenshot('cancel-modal.png', { maxDiffPixels: 100 });
});
```

This dovetails with the existing `design-clone` / `design-visual-qa` skills if they're available.

## When to push back instead of generating

- The YAML lists >5 e2e tests → "this many e2e tests will be slow and flaky. Consider moving the variants to integration; e2e should be the single critical journey."
- The journey requires complex test data setup (multiple users, prior history, third-party integrations) → suggest splitting: keep the journey at e2e, but cover the data variations at integration.
- The project has no UI / API at all (pure library) → e2e doesn't apply; emit nothing for this layer, note in the README.
