# Sub-step 4-E: Mutation testing config

**Goal:** emit a tool-specific mutation-testing config and a `mutation.sh` runner that enforces the kill-rate threshold from `done_when.yaml`.

**Background:** see `../anti-cheating-mutation.md` for why this is mandatory.

**Do NOT run the mutation suite during generation.** It's minutes-to-hours. Just emit and document.

---

## Python — mutmut

Add to `pyproject.toml` (or create one if missing):

```toml
[tool.mutmut]
paths_to_mutate = "src/"
runner = "pytest -x tests/<feature>/unit/"
tests_dir = "tests/<feature>/unit/"
backup = false
use_coverage = true
```

Notes:
- `runner = "pytest -x"` stops on first fail per mutant — faster.
- `use_coverage = true` skips mutants on lines no test covers (those would survive trivially and pollute kill rate).

Runner script (`tests/<feature>/mutation.sh`):

```bash
#!/usr/bin/env bash
set -euo pipefail

# Generated by test-suite-generator/<skill-version>   # substitute from SKILL.md frontmatter `version:`
# Threshold pulled from done_when.yaml: mutation_kill_rate >= 0.70

THRESHOLD=70  # percent

mutmut run
RESULT=$(mutmut results | tail -1)
# mutmut output format: "killed: N, survived: M, ..."
KILLED=$(echo "$RESULT" | grep -oP 'killed:\s*\K\d+')
SURVIVED=$(echo "$RESULT" | grep -oP 'survived:\s*\K\d+')
TOTAL=$((KILLED + SURVIVED))
if [[ $TOTAL -eq 0 ]]; then
  echo "ERROR: no mutants generated — likely a path/runner config issue."
  exit 1
fi
SCORE=$(( KILLED * 100 / TOTAL ))
echo "Kill rate: $SCORE% ($KILLED / $TOTAL)"
if [[ $SCORE -lt $THRESHOLD ]]; then
  echo "✗ Below threshold $THRESHOLD%"
  echo "Surviving mutants:"
  mutmut results | grep -E '^(survived|timeout)' | head -20
  exit 1
fi
echo "✓ Kill rate above $THRESHOLD%"
```

## TypeScript — Stryker

`stryker.conf.json`:

```json
{
  "$schema": "./node_modules/@stryker-mutator/core/schema/stryker-schema.json",
  "packageManager": "npm",
  "testRunner": "vitest",
  "mutate": [
    "src/<feature-module>/**/*.ts",
    "!src/**/*.test.ts",
    "!src/**/*.spec.ts",
    "!src/**/index.ts",
    "!src/**/types.ts"
  ],
  "coverageAnalysis": "perTest",
  "thresholds": { "high": 80, "low": 70, "break": 70 },
  "reporters": ["progress", "clear-text", "html"]
}
```

Critical:
- `"break": 70` — Stryker exits nonzero if mutation score drops below 70%. This is the gate.
- `coverageAnalysis: "perTest"` — only runs the tests that cover each mutant. Massive speedup.
- Excluded paths: types and index re-exports — these have no logic to mutate; including them pollutes the denominator.
- **`mutate:` uses ONE inclusive glob per source root, not a glob plus the specific file it already contains.** Listing `"src/billing/cancel_subscription_use_case.ts"` *and* `"src/billing/**/*.ts"` is redundant — the second entry already includes the first. Pick whichever scope the feature actually occupies (typically the per-feature module glob, e.g. `src/billing/**/*.ts` if all the feature's source lives in `src/billing/`) and rely on the negation patterns below to trim test files and DTOs. Mixing specific paths with broad globs is a P2 cleanliness bug and confuses anyone auditing the mutate scope.

Runner script (`tests/<feature>/mutation.sh`):

```bash
#!/usr/bin/env bash
set -euo pipefail
npx stryker run --reporters progress,clear-text
# Stryker exits 1 automatically if score < break threshold (70)
```

## Kotlin / Java — PIT

In `build.gradle.kts`:

```kotlin
plugins {
  id("info.solidsoft.pitest") version "1.15.0"
}

pitest {
  targetClasses.set(setOf("com.example.${project.name}.*"))
  mutators.set(setOf("STRONGER"))
  threads.set(4)
  outputFormats.set(setOf("XML", "HTML"))
  mutationThreshold.set(70)   // ← the gate
  coverageThreshold.set(80)
  testPlugin.set("junit5")
}
```

Runner: `./gradlew pitest`. Build fails if `mutationThreshold` not met.

## Swift — Muter

`muter.conf.yml`:

```yaml
executable: xcodebuild
arguments:
  - test
  - -scheme
  - <YourScheme>
exclude:
  - "*.generated.swift"
  - "Tests/*"
```

Run: `muter`. Inspect `muter-report.html`; Swift mutation testing is less mature, so enforce the threshold in a separate script:

```bash
SCORE=$(jq '.mutationScore' muter-report.json)
[[ "$SCORE" -ge 70 ]] || { echo "Swift mutation score $SCORE < 70"; exit 1; }
```

## README addendum for the mutation script

Append to the project's testing README:

```markdown
## Mutation testing

Mutation tests run via `bash tests/<feature>/mutation.sh`. They take time
(minutes to hours depending on project size) and so are typically run pre-merge
in CI or nightly, not in the inner dev loop.

The kill-rate threshold (70%) is **not optional**. It is the anti-reward-hacking
gate from done_when.yaml. If kill rate drops, look at the surviving mutants —
they pinpoint exactly which assertions are missing from the suite.

If you find yourself wanting to lower the threshold, instead:
1. Look at the surviving mutants — usually they reveal a real test gap.
2. If a file is genuinely un-mutable (pure DTOs, generated code), exclude it
   in the mutation tool config — don't lower the global threshold.
```

## Common config pitfalls

| Pitfall | Fix |
|---|---|
| Running mutation across the entire `src/` while only generating tests for one feature → kill rate is artificially low | Scope `paths_to_mutate` / `mutate:` to the files this feature touches |
| Including generated files (proto, GraphQL clients) → meaningless mutants | Exclude them explicitly |
| Not setting a runner that fails fast on first test failure | Use `-x` (pytest) or equivalent; saves 10x time |
| Running mutation in the same process as the rest of the suite | Always run in CI as a separate job; long-running tests should not block dev feedback |
| Setting threshold too low ("any number is fine") | The whole point is the threshold. 70 is the floor; don't go below without a documented reason |
