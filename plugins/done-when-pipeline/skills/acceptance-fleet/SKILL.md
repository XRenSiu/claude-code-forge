# acceptance-fleet — Step 5-6 multi-agent acceptance loop

You are invoked to verify an implementation against a `done_when.yaml` contract via a fleet of independent specialized evaluators, aggregate their findings through a non-debating meta-judge, and decide the next ratchet state (DONE / FIX / SPEC_DRIFT / GAMING_RISK).

**Say once at the start, then start working:**
> "I'm using the acceptance-fleet skill. I'll spawn the 7-role evaluation fleet + spec-gaming-detector in parallel, aggregate via meta-judge (no debate), and emit a four-state ratchet decision + a full ratchet-log/iteration-NNN/ trace."

Do not narrate further — just walk the phases.

---

## Iron rules (re-read before every iteration)

1. **Implementation agent MUST NOT see evaluator prompts.** This is the single highest-leverage anti-gaming guarantee. Evaluator prompt leakage = gaming substrate (per MD §6 "一定不要做"). Structurally enforce one of four isolation levels (see `references/evaluation-isolation-levels.md`); minimum acceptable is the "medium" level (different model sizes, no shared session). If you cannot guarantee at least medium isolation, **refuse the run** and tell the user why — do not proceed and pretend.
2. **No agent-to-agent debate.** *Judging with Many Minds* (Dartmouth/Yale 2025) shows debate frameworks amplify position / verbosity / CoT / bandwagon bias after round 1, and continue amplifying. Meta-judge frameworks resist bias. The 7 fleet roles spawn independently, give findings independently, and are aggregated by a meta-judge that does **not** re-review the code — it only synthesizes the findings list. **Never let two evaluator agents see each other's outputs while still forming opinions.**
3. **Model diversity > evaluator count.** Two cross-vendor evaluators (Claude + Codex, or Claude + Gemini) beat five same-vendor agents (Milvus benchmark, MD §1.4). Always include at least one non-Claude evaluator for the adversarial-reviewer role. If the runtime only has Claude available, use mixed sizes (Haiku + Sonnet + Opus) — same RLHF pipeline but different capability distributions; weaker than cross-vendor but better than all-Opus.
4. **LLM judges generation-quality, programs judge correctness.** *Rethinking LLMs as Verifiers* (OpenReview 2026) splits the solver-verifier gap by task type — LLMs verify generation-quality (clarity, intent match) *better* than they solve; LLMs verify correctness (does the code do X) *worse* than they solve. Route accordingly: PBT / mutation / testcontainers for correctness; LLM evaluators for design intent, doc quality, persona-fit. **Never route correctness through an LLM judge** when a programmatic check exists.
5. **Every finding carries evidence; rebuttable findings are dropped.** Borrowed from Anthropic Code Review's "verification step" (MD §5.2): before a finding ships, a separate agent tries to rebut it. Successful rebuttals = drop the finding. Anthropic data: false positive rate fell from double-digit % to <1% with this gate. The fleet must run this rebuttal pass *before* meta-judge aggregation, not after.
6. **Meta-judge does NOT re-review code.** It only synthesizes findings: dedupe by `file:line + root_cause`, weight by source diversity (multi-source + cross-vendor = boost; single-source + single-vendor = penalty), check against `done_when.yaml` thresholds, surface conflicts as `NEEDS_HUMAN`. The moment meta-judge starts forming its own opinions about the code, it introduces its own bias and stops being a judge — it becomes another reviewer. Hard wall.
7. **4-Eyes Principle.** The agent that flagged a finding as suspicious cannot be the agent that confirms or refutes it. The rebuttal pass (rule 5) MUST use a different evaluator than the one that originated the finding. Trivially enforce: rebuttal pass uses adversarial-reviewer's cross-vendor sibling; if adversarial-reviewer originated the finding, rebuttal uses Claude requirement-tracer.
8. **Ratchet is four-state, not two.** `DONE / FIX / SPEC_DRIFT / GAMING_RISK`. Specifically:
   - `DONE` — all thresholds pass AND `gaming_risk_score < 3` AND no `surfaced_vector` from `spec-robustness.md` triggered.
   - `FIX` — partial pass with clear findings; emit fix-prompt and feed back to implementation agent (a separate session — see rule 1). Increment iteration counter.
   - `SPEC_DRIFT` — PBT counterexamples consistent with REQ text recur across N iterations (default N from `spec_drift_threshold.max_fix_loops_before_escalation`, fallback 3), OR `surfaced_vector` from `spec-robustness.md` repeatedly triggered with no fix path. **Do not patch code.** Return control to `/acceptance-spec` to narrow the REQ.
   - `GAMING_RISK` — `gaming_risk_score >= 7` from spec-gaming-detector. **Do not patch code.** Trigger spec strengthening: list the gaming vectors that landed, suggest concrete `done_when.yaml.behavior.thresholds:` adjustments or REQ rewrites, hand back to `/acceptance-spec`.
9. **Every iteration writes a complete ratchet-log entry.** Persistence is not optional — it is the substrate for debugging, training-data reuse, and audit (ISO 26262 / SOC 2). Empty entries on no-op iterations are still written. See `references/ratchet-log-format.md` for the directory layout.
10. **`spec-robustness.md` is a first-class input.** Its `surfaced_vectors` block tells spec-gaming-detector what to specifically watch for in this feature. Its `verifier_hints` block carries domain-specific gaming pre-warnings from `/acceptance-spec` S2.5. If `spec-robustness.md` is missing, S2.5 didn't run; **set spec-gaming-detector to maximum suspicion mode** and emit a warning that the upstream contract is incomplete. Do not refuse the run, but the warning surfaces.
11. **Findings have a single schema regardless of which evaluator emitted them.** `references/finding-schema.yaml` is the contract. Each evaluator output is a YAML list conforming to that schema. Evaluators that try to invent freeform output get re-prompted once with the schema; second failure = drop their batch and proceed without (better silent partial pass than corrupted aggregation).
12. **Spec-gaming-detector runs every iteration, not just iteration 1.** Gaming patterns can be introduced mid-loop (an implementation agent under FIX pressure starts weakening assertions). The detector watches the *diff* between iterations, not just the absolute state. See `references/gaming-detector-patterns.md`.

If you catch yourself routing a correctness check through an LLM judge, **stop and find the programmatic equivalent**. Rule 4 violations are the most common silent failure of this skill.

## Phase map

```
S0  Bootstrap         read specs/<feature>/, validate inputs, detect evaluator availability, set isolation level
S1  Fleet spawn       7 evaluators + spec-gaming-detector run in parallel, all reading the same impl artifacts
S2  Rebuttal pass     each finding is challenged by a different evaluator before reaching meta-judge
S3  Meta-judge        dedupe, weight, check thresholds, classify state — does not re-review code
S4  Ratchet decision  DONE | FIX | SPEC_DRIFT | GAMING_RISK; emit fix-prompt if FIX
S5  Persist           write ratchet-log/iteration-NNN/ — full trace, no truncation
```

The user sees one summary line per iteration during execution and a final ratchet-state decision. The full trace is on disk.

## S0 — Bootstrap

1. Parse `$ARGUMENTS`. Must resolve to a `specs/<feature>/` directory containing **at minimum**: `spec.md`, `done_when.yaml`, `spec-robustness.md`, and a sibling `tests/<feature>/` directory (generated by `/test-suite-generator`). If `spec-robustness.md` is missing, set the warning flag (see iron rule 10) but proceed.
2. Detect the **implementation root** (`src/` typically). Confirm with the user only if ambiguous (e.g. monorepo with multiple language roots).
3. Determine **isolation level** by checking what evaluator runtimes are available:
   - Codex CLI present (`which codex` succeeds) → `strong` (cross-vendor available).
   - Only Claude available, but `claude-haiku-4-5` / `claude-sonnet-4-6` / `claude-opus-4-7` selectable → `medium`.
   - Single-model-only environment → `weak` (warn user — barely meets minimum).
   - No way to run a second session without sharing context → **refuse** and explain (see iron rule 1).

   Record the chosen level in `ratchet-log/iteration-NNN/isolation.json`.

4. Determine **iteration number** by scanning existing `ratchet-log/` entries. If none, iteration 1. Otherwise next sequential.
5. Read `done_when.yaml.spec_drift_threshold.max_fix_loops_before_escalation` (default 3 if absent). Store as `SPEC_DRIFT_TRIGGER`.
6. Read `spec-robustness.md.surfaced_vectors` and `spec-robustness.md.verifier_hints`. These become inputs to spec-gaming-detector's prompt.

No user output yet beyond a one-line "starting iteration N with `<level>` isolation".

## S1 — Fleet spawn (parallel, independent)

Spawn 8 evaluators in parallel. Each reads the SAME impl artifacts (`src/` + `tests/<feature>/` + `spec.md` + `done_when.yaml`) but no two share a session. Each writes its finding list (per `references/finding-schema.yaml`) into a per-role output file under `ratchet-log/iteration-NNN/fleet-outputs/`.

Reference: `references/role-prompts.md` carries the prompt template for each role. Do NOT inline the prompts here; the indirection is what makes prompt iteration tractable.

### The 8 roles

| Role | Model (preferred) | Model (medium-isolation fallback) | What it produces |
|---|---|---|---|
| `test-runner` | Haiku | Haiku | Runs `tests/<feature>/` via the test-suite-generator manifest. Outputs raw pass/fail/coverage/mutation-kill numbers. No findings — just the run report. |
| `existence-checker` | Haiku | Haiku | Runs `tests/<feature>/existence.sh`. Verifies every entry in `done_when.yaml.existence:` resolves in `src/`. Outputs binary checks list. |
| `requirement-tracer` | Opus + LOCATE/READ/RETRIEVE tools | Opus | Agent-as-Judge pattern (DevAI ICML 2025). For each REQ-ID in `spec.md`, use `LOCATE` (glob/grep), `READ` (read_file), `RETRIEVE` (find matching test) to gather evidence. Per-REQ verdict: `fully_compliant` / `partially_compliant` / `not_compliant` / `requires_human_verification`. |
| `design-reviewer` | Opus | Sonnet | Checks impl matches spec *intent*, not just literal SHALL clauses. Flags "form satisfied, spirit violated" cases. |
| `adversarial-reviewer` | **Codex or Gemini (cross-vendor)** | Sonnet (different size from impl) | Reversed-default prompt (MD §3.3 γ theory). "Assume this code caused a production incident. Find why. If you say 'looks good', you fail." Severity P0-P3 with reproduction. |
| `edge-case-hunter` | Opus | Opus | Systematic path tracing for unhandled boundaries, races, exception paths. Borrowed from BMAD `edge-case-hunter`. |
| `e2e-explorer` | Opus + Playwright MCP | Opus + Playwright MCP | Drives the real UI / API like a user. Auto-derives happy path + key edge cases from `spec.md` rather than running prewritten Playwright scripts. Borrowed from Builder.io QRA. |
| `spec-gaming-detector` | Opus + git_diff tools | Opus | Reads `spec-robustness.md.surfaced_vectors`, scans impl for the six RHD patterns + any domain-specific hints, computes `gaming_risk_score: 0-10`. Watches the impl diff between iterations, not just absolute state. |

**Output discipline.** Each role writes exactly one file: `ratchet-log/iteration-NNN/fleet-outputs/<role>.yaml`. Files conform to `references/finding-schema.yaml`. Roles that produce no findings emit `findings: []` plus their meta block — they do not skip the file.

**Parallelism note.** All 8 spawn in the same dispatch. Do NOT serialize "fast roles first, slow roles after" — that just makes the wall-clock worse without giving any role new information.

## S2 — Rebuttal pass

For each finding emitted in S1, route it to a *different* evaluator's session for a one-shot rebuttal attempt. The 4-Eyes Principle (iron rule 7): the originating role cannot be the rebutter.

Rebuttal routing table:

| Originating role | Rebutter role |
|---|---|
| `adversarial-reviewer` (cross-vendor) | `requirement-tracer` (Claude Opus) |
| `requirement-tracer` | `adversarial-reviewer` |
| `design-reviewer` | `edge-case-hunter` |
| `edge-case-hunter` | `design-reviewer` |
| `e2e-explorer` | `requirement-tracer` |
| `spec-gaming-detector` | `adversarial-reviewer` (cross-vendor) |
| `test-runner` / `existence-checker` | (no rebuttal — these are programmatic outputs, not LLM judgments) |

The rebuttal prompt is fixed: "Here is a finding from `<role>`. Try to rebut it. If you can construct a plausible scenario where the finding is wrong (false positive), state that scenario and what evidence would confirm it. If you cannot, state 'finding stands' and stop."

Findings that get successfully rebutted are moved to `rebutted_findings:` in the per-role output file (not deleted — kept for audit). Findings that stand are passed to S3.

**Do not run more than one rebuttal pass.** Multiple rounds invite the debate dynamics rule 2 forbids. If a finding is contested but evidence is ambiguous, it goes to meta-judge as-is and may be classified `NEEDS_HUMAN` in S3.

## S3 — Meta-judge (synthesis only, no re-review)

Spawn a **single** meta-judge session (Opus 4.7 — never weaker). Input: every role's surviving findings + the rebutted list (for context) + `done_when.yaml` + `spec-robustness.md`.

Meta-judge has exactly four actions (see iron rule 6 hard wall):

1. **Dedupe.** Two findings are duplicates if they reference the same `file:line` AND have semantically equivalent `root_cause`. Multi-source surviving duplicates get a `confidence_boost: +0.2` per additional independent source (capped at +0.4). Cross-vendor duplicates double the boost.
2. **Conflict arbitration.** Two findings on the same `file:line` with contradictory verdicts (one says "bug here", another says "behavior correct") → look at evidence strength, not vote count. If evidence is clearly stronger on one side, take that side; if ambiguous, classify the finding as `NEEDS_HUMAN`.
3. **Threshold check.** For each finding, determine whether it triggers a `done_when.yaml.behavior.thresholds:` violation, a `spec-robustness.md.surfaced_vector` watch, or stands as a freeform observation. Tally the gaming_risk_score from spec-gaming-detector.
4. **State classification.** Apply the four-state ratchet rules (iron rule 8) and write the decision into `ratchet-log/iteration-NNN/meta-judge-output.yaml`.

Meta-judge output schema is fixed (see `references/finding-schema.yaml` § "meta-judge output"). The single most important field is `state_decision: DONE | FIX | SPEC_DRIFT | GAMING_RISK | NEEDS_HUMAN`.

`NEEDS_HUMAN` is the fifth state added by meta-judge alone — it signals "the four-state machine cannot decide; a human must look." Used when:
- A `NEEDS_HUMAN`-classified finding from conflict arbitration cannot be safely auto-resolved.
- `requirement-tracer` reports `requires_human_verification` on a REQ (UI/UX intent the LLM cannot judge).
- Cross-vendor evaluators disagree on a P0/P1 finding with comparable evidence strength.

**Never let meta-judge silently auto-resolve `NEEDS_HUMAN`** — that defeats its purpose.

## S4 — Ratchet decision

Based on `meta-judge-output.yaml.state_decision`:

### `DONE`
1. Write `ratchet-log/iteration-NNN/final-state.json` with the verdict.
2. Tell the user: "Iteration N complete. All thresholds pass. gaming_risk_score = X. <K> findings rebutted, <M> findings classified non-blocking, 0 blocking. Acceptance fleet exits."
3. End the skill.

### `FIX`
1. Generate the fix-prompt from the blocking findings (see `references/fix-prompt-template.md`). Each blocking finding becomes one bullet in the prompt with `file:line + root_cause + evidence + minimal-change-suggestion`. The prompt is hand-off material for the implementation agent's *next session* (rule 1: same session forbidden).
2. Write `ratchet-log/iteration-NNN/fix-prompt.md`.
3. Tell the user: "Iteration N → FIX. <K> blocking findings. fix-prompt at `ratchet-log/iteration-N/fix-prompt.md`. Feed this to a fresh implementation session, then re-invoke `/acceptance-fleet` for iteration N+1."
4. End the skill — do NOT auto-spawn the implementation agent. That would violate isolation if mishandled; the user explicitly hands off so they can pick which implementer and confirm isolation.

### `SPEC_DRIFT`
1. Write the offending REQs + the PBT counterexamples to `ratchet-log/iteration-NNN/spec-drift-report.md`.
2. Tell the user: "Iteration N → SPEC_DRIFT. <K> REQs have produced PBT counterexamples consistent with their literal text across <M> consecutive iterations. The spec is likely the bug, not the code. Re-invoke `/acceptance-spec specs/<feature>/` with these REQs as the narrow brief; the clarify loop will tighten them. Do NOT continue patching code."
3. End the skill.

### `GAMING_RISK`
1. Write the gaming vectors that landed + suggested `done_when.yaml` / `spec.md` adjustments to `ratchet-log/iteration-NNN/gaming-risk-report.md`.
2. Tell the user: "Iteration N → GAMING_RISK. gaming_risk_score = X (threshold 7). Patterns observed: <list>. The contract has structural gaps the implementer is exploiting. Re-invoke `/acceptance-spec` with this report as the brief; S2.5 will close the surfaced vectors. Do NOT continue patching code."
3. End the skill.

### `NEEDS_HUMAN`
1. Write the deadlock-causing finding(s) to `ratchet-log/iteration-NNN/needs-human.md` with the specific question(s) requiring human judgment.
2. Tell the user: "Iteration N → NEEDS_HUMAN. Meta-judge cannot decide. Human input required for: <list of questions>. Answer in your next message; I'll then re-classify the state."
3. **Do not** end the skill — wait for user input, then resume at S3 with the answers folded into the meta-judge state classification.

## S5 — Persist (always runs, regardless of state)

Every iteration MUST leave a complete `ratchet-log/iteration-NNN/` directory. See `references/ratchet-log-format.md` for the full layout. At minimum:

```
ratchet-log/iteration-NNN/
├── timestamp.txt
├── isolation.json
├── input-manifest.json          # which spec/test files, with checksums
├── impl-diff.patch              # impl agent's changes this iteration (if known)
├── fleet-outputs/
│   ├── test-runner.yaml
│   ├── existence-checker.yaml
│   ├── requirement-tracer.yaml
│   ├── design-reviewer.yaml
│   ├── adversarial-reviewer.yaml
│   ├── edge-case-hunter.yaml
│   ├── e2e-explorer.yaml
│   └── spec-gaming-detector.yaml
├── rebuttals/
│   └── <finding-id>.yaml        # one per challenged finding, regardless of outcome
├── meta-judge-output.yaml
├── final-state.json             # the state decision + summary
└── <state-specific report>.md   # fix-prompt.md | spec-drift-report.md | gaming-risk-report.md | needs-human.md
```

This directory is the contract — anything downstream (debugging, training, audit) reads from here. Never truncate, never compress, never delete iteration directories from within the skill.

## When to refuse / redirect

- **Inputs incomplete.** Missing `spec.md` / `done_when.yaml` → refuse, tell the user to run `/acceptance-spec` first. Missing `tests/<feature>/` → refuse, tell the user to run `/test-suite-generator` first.
- **Missing `spec-robustness.md`.** Do NOT refuse — proceed with a warning that S2.5 was skipped and spec-gaming-detector is on maximum suspicion. The output will note "spec-robustness.md absent — verifier ran in max-suspicion mode".
- **Cannot guarantee at least medium isolation.** Refuse. Explain that running in a single shared session means the impl agent can read the evaluator's prompt next iteration, which destroys the whole anti-gaming structure. Suggest: open a fresh Claude session for the implementation handoff, even within Claude Code.
- **No tests can run** (testcontainers not available, Playwright not installed, etc.). Partial-run: skip the layer(s) that can't run, mark their evaluators' outputs as `skipped: <reason>`, proceed with the rest. Meta-judge classifies as `NEEDS_HUMAN` for any threshold that depends on a skipped layer.
- **`done_when.yaml` schema violation.** Refuse. The contract must conform to Appendix C v1 — this skill does not attempt to repair contracts. Send the user back to `/acceptance-spec`.

## Cost expectation (per iteration)

One iteration runs all 8 evaluators + rebuttals + meta-judge. Rough order of magnitude with prompt caching:

| Stage | Tokens (rough) | Cost (rough) |
|---|---|---|
| 8 evaluators in parallel | ~150K in / 25K out total | ~$0.55 |
| Rebuttal pass (one per ~5-10 findings) | ~30K in / 5K out | ~$0.10 |
| Meta-judge | ~25K in / 4K out | ~$0.10 |
| **Per-iteration total** | | **~$0.75** |

Pipelines usually converge in 2-4 iterations (the four-state machine catches spec-drift and gaming-risk early instead of looping FIX endlessly). Budget ~$2-3 per feature acceptance.

Cross-vendor evaluators cost more per call but converge faster (fewer iterations because cross-vendor catches more bugs per iteration). Net cost usually lower than same-vendor-only despite higher per-call cost.
