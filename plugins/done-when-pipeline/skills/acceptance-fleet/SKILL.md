# acceptance-fleet — Step 5-6 orchestrator (v1.0+ pure dispatch)

You are invoked to verify an implementation against a `done_when.yaml` contract by dispatching the 6 standalone review skills, synthesizing their findings via `/meta-judge`, and decoding the verdict into a four-state ratchet decision (DONE / FIX / SPEC_DRIFT / GAMING_RISK).

**Say once at the start, then start working:**
> "I'm using the acceptance-fleet skill. I'll dispatch the 6 review skills in parallel against your impl, hand findings to /meta-judge, decode the verdict into the 4-state ratchet, and write a full ratchet-log/iteration-NNN/ trace."

Do not narrate further — just walk the phases.

---

## What changed in v1.0.0 (read this once if migrating from v0.x)

| v0.x (monolithic fleet) | v1.0+ (orchestrator) |
|---|---|
| 7 evaluator roles embedded internally (test-runner / existence-checker / requirement-tracer / design-reviewer / adversarial-reviewer / edge-case-hunter / e2e-explorer) | 6 standalone skills dispatched: `/code-reviewer` × focuses, `/qa-reviewer`, `/pm-reviewer`, `/spec-drift-detector`, `/spec-gaming-detector` |
| Meta-judge was a fleet phase (S3) | `/meta-judge` is its own skill, invoked by this orchestrator |
| `references/role-prompts.md` carried all 8 role prompts | Each review skill owns its prompt + references; orchestrator just calls them |
| `references/finding-schema.yaml` had all schemas | Each review skill owns its own finding-schema.yaml; meta-judge owns the final-verdict schema |
| `gaming-detector-patterns.md`, `meta-judge-protocol.md` lived in this skill | Migrated to `spec-gaming-detector` and `meta-judge` respectively |
| `done_when.yaml` had a `fitness:` layer | Retired per HTML v2 §3.5; replaced with `rules:` layer consumed by `/meta-judge` |

If you have v0.x `ratchet-log/` directories, they remain readable — the new orchestrator writes the same format under `references/ratchet-log-format.md`. The migration is at the *skill definition* level, not the artifact level.

## Iron rules (retained at orchestrator level)

The pre-v1.0 fleet had 12 iron rules. Most of them migrated into the individual review skills (where they belong — rules about adversarial-reviewer's prompt now live in `/code-reviewer --adversarial`; rules about meta-judge's hard wall now live in `/meta-judge`). The five rules below are the ones the *orchestrator* still owns:

1. **Implementation agent MUST NOT see evaluator prompts.** Single highest-leverage anti-gaming guarantee. Structurally enforce one of four isolation levels (see `references/evaluation-isolation-levels.md`); minimum acceptable is "medium" (different model sizes, no shared session). If you cannot guarantee at least medium isolation, **refuse the run** — do not proceed and pretend.
2. **No agent-to-agent debate.** *Judging with Many Minds* (Dartmouth/Yale 2025). The 6 review skills spawn independently, give findings independently, are aggregated by `/meta-judge` which does NOT re-review. Never let two review skills see each other's outputs while still forming opinions. This is the responsibility of THIS orchestrator — `/meta-judge` enforces the hard wall on its side, but acceptance-fleet enforces the parallel-spawn no-cross-talk on its side.
3. **Ratchet is four-state, not two.** `DONE / FIX / SPEC_DRIFT / GAMING_RISK`:
   - `DONE` — `/meta-judge` returned PASS AND `gaming_risk_score < 3` AND no `surfaced_vector` from `spec-robustness.md` triggered.
   - `FIX` — `/meta-judge` returned BLOCK_MERGE with clear findings; emit fix-prompt and feed back to implementation agent (a separate session — see rule 1). Increment iteration counter.
   - `SPEC_DRIFT` — PBT counterexamples consistent with REQ text recur across N iterations (default N from `spec_drift_threshold.max_fix_loops_before_escalation`, fallback 3), OR `surfaced_vector` repeatedly triggered with no fix path, OR `/spec-drift-detector` reports recurring drift signals. **Do not patch code.** Return control to `/acceptance-spec` to narrow the REQ.
   - `GAMING_RISK` — `gaming_risk_score >= 7` from `/spec-gaming-detector`. **Do not patch code.** Trigger spec strengthening: list the gaming vectors that landed, hand back to `/acceptance-spec`.
   - When `/meta-judge` returns NEEDS_HUMAN, the orchestrator surfaces the question and waits — does NOT auto-pick a ratchet state.
4. **Every iteration writes a complete ratchet-log entry.** Persistence is not optional — it is the substrate for debugging, training-data reuse, and audit (ISO 26262 / SOC 2). Empty entries on no-op iterations are still written. See `references/ratchet-log-format.md` for the directory layout.
5. **`spec-robustness.md` is a first-class input.** Pass through to `/spec-gaming-detector` via `--spec-robustness=<path>`. If `spec-robustness.md` is missing, S2.5 didn't run; warn the user, set `/spec-gaming-detector` to maximum suspicion mode (no `--spec-robustness` arg → its iron rule 10 takes over).

The other rules from v0.x (cross-vendor preference, rebuttal step, schema discipline, gaming-detector-every-iteration) are now enforced *inside* the review skills. The orchestrator trusts them.

## Inputs

| Arg | Required | Notes |
|---|---|---|
| `<spec_dir>` | yes | Path to `specs/<feature>/`. Must contain `spec.md`, `done_when.yaml`, optionally `spec-robustness.md`, and a sibling `tests/<feature>/` (generated by `/test-suite-generator`). |
| `--impl-root=<path>` | no, inferred | Implementation root. Defaults to `src/` next to the spec dir. |

| `--max-iterations=<int>` | no, default 5 | Hard cap on FIX-state iterations before forced human escalation. |

## Phase map

```
S0  Bootstrap         read specs/<feature>/, validate inputs, detect isolation, set iteration number
S1  Dispatch fleet    spawn 6 review skills in PARALLEL against the impl + tests + spec
S2  Synthesize        invoke /meta-judge with all 6 review outputs + rules source
S3  Ratchet decision  decode meta-judge verdict + gaming/drift signals into DONE | FIX | SPEC_DRIFT | GAMING_RISK | NEEDS_HUMAN
S4  Emit state report DONE: archive; FIX: write fix-prompt; SPEC_DRIFT: drift report; GAMING_RISK: gaming report; NEEDS_HUMAN: wait
S5  Persist           write full ratchet-log/iteration-NNN/ trace
```

The skill exits at S4 in DONE / FIX / SPEC_DRIFT / GAMING_RISK states. NEEDS_HUMAN waits for user input then re-enters S3 with the answer folded in.

---

## S0 — Bootstrap

1. Parse `$ARGUMENTS`. Must resolve to a `specs/<feature>/` directory containing at minimum: `spec.md`, `done_when.yaml`, and a sibling `tests/<feature>/`. If `spec-robustness.md` is missing, set the warning flag (iron rule 5) but proceed.
2. Detect `--impl-root`. Confirm with user only if ambiguous (multi-language monorepo).
3. Determine isolation level (see `references/evaluation-isolation-levels.md` for the full algorithm):
   - `which codex` succeeds → `strong` (cross-vendor available)
   - Only Claude available, but multiple sizes selectable → `medium`
   - Single-model-only environment → `weak` (warn user)
   - No way to spawn fresh sessions → **refuse** with explanation
   Record in `ratchet-log/iteration-NNN/isolation.json`.
4. Determine iteration number by scanning existing `ratchet-log/` entries. None → iteration 1. Otherwise next sequential.
5. Read `done_when.yaml.spec_drift_threshold.max_fix_loops_before_escalation` (default 3). Store as `SPEC_DRIFT_TRIGGER`.
6. Read `spec-robustness.md` if present.

One-line user output: "acceptance-fleet: starting iteration N with `<level>` isolation; <6 skills will be dispatched>".

## S1 — Dispatch fleet (parallel, independent)

Spawn the 6 review skills **in parallel**. No two share a session. Each writes its output file under `ratchet-log/iteration-NNN/fleet-outputs/`. See `references/skill-dispatch-matrix.md` for the exact command + arguments per skill, and how the previous v0.x roles map to v1.0+ skill invocations.

Brief overview:

| Skill | Args | Output file |
|---|---|---|
| `/code-reviewer --focus=security --adversarial` | impl diff (HEAD~1..HEAD if available) | `code-reviewer-security.yaml` |
| `/code-reviewer --focus=logic` | impl diff | `code-reviewer-logic.yaml` |
| `/code-reviewer --focus=perf` (cheap layer) | impl diff | `code-reviewer-perf.yaml` |
| `/qa-reviewer` | `tests/<feature>/`, `--thresholds=<spec_dir>/done_when.yaml`, `--baseline=<prev iteration>` | `qa-reviewer.yaml` |
| `/pm-reviewer` | `<spec_dir>/spec.md`, `<impl-root>`, `--severity-marks=<critical REQs>` | `pm-reviewer.yaml` |
| `/spec-drift-detector` | `<spec_dir>/spec.md`, `<impl-root>` | `spec-drift-detector.yaml` |
| `/spec-gaming-detector` | `<spec_dir>/spec.md`, `<impl-root>`, `--spec-robustness=<spec_dir>/spec-robustness.md` (if present), `--history=<prev iteration>/impl-snapshot.tar.gz` | `spec-gaming-detector.yaml` |

**Parallelism note.** All 7 dispatch calls (3 code-reviewer focuses + qa/pm/drift/gaming) happen in the same dispatch. Do NOT serialize "fast first, slow after" — that wastes wall clock without helping any skill see new information.

**Cross-vendor allocation.** If `strong` isolation is active (Codex / Gemini CLI available):
- `code-reviewer --adversarial` runs on the cross-vendor evaluator.
- `spec-gaming-detector` runs on the cross-vendor evaluator.
- Other skills stay on Claude (with size mixing — Haiku for cheap parts, Opus for reasoning).

If only `medium` isolation, all skills use Claude but with mixed sizes (qa-reviewer on Sonnet, others on Opus, classifier sub-agents on Haiku). The skills internally handle their model choice; the orchestrator only sets the cross-vendor hint via per-skill flags.

## S2 — Synthesize via /meta-judge

Invoke `/meta-judge` with:

```
/meta-judge <reviews_source=ratchet-log/iteration-NNN/fleet-outputs/> \
            --rules=<spec_dir>/done_when.yaml \
            --context='{"feature":"<feature_name>","iteration":N,"is_hotfix":false}'
```

`/meta-judge` writes `meta-judge-output.yaml` to `ratchet-log/iteration-NNN/`. The orchestrator reads this back for S3.

`/meta-judge` enforces its own hard wall (no re-reviewing code) — the orchestrator trusts that.

---

## S3 — Ratchet decision

Decode meta-judge's `state_decision` plus additional signals from `/spec-gaming-detector` and `/spec-drift-detector` into one of the four ratchet states (plus NEEDS_HUMAN escape).

Algorithm (strict order, first match wins):

```
A. /meta-judge state_decision == NEEDS_HUMAN
   → ratchet state: NEEDS_HUMAN
   → surface the needs_human_items, wait for user input, re-run S3.

B. /spec-gaming-detector gaming_risk_score >= 7
   OR gaming_risk_trajectory shows monotonic growth ≥ 2 per iteration for 2+ iterations
   OR a surfaced_vector triggered for 2+ consecutive iterations
   → ratchet state: GAMING_RISK
   → emit gaming-risk-report.md; do NOT generate fix-prompt; hand back to /acceptance-spec.

C. /spec-drift-detector signals recurring "behavior_mismatch" with same REQ-ID across SPEC_DRIFT_TRIGGER iterations
   OR /pm-reviewer "not_compliant" on same REQ across SPEC_DRIFT_TRIGGER iterations despite impl changing
   → ratchet state: SPEC_DRIFT
   → emit spec-drift-report.md; do NOT generate fix-prompt; hand back to /acceptance-spec.

D. /meta-judge state_decision == BLOCK_MERGE
   → ratchet state: FIX
   → generate fix-prompt.md from meta-judge's blocking deduplicated_findings (cap 8 per references/fix-prompt-template.md).

E. /meta-judge state_decision == PASS
   AND /spec-gaming-detector gaming_risk_score < 3
   AND no recurring drift or gaming patterns from B/C
   → ratchet state: DONE
   → archive iteration; tell user the loop is complete.
```

`SPEC_DRIFT_TRIGGER` defaults to 3 (from `done_when.yaml.spec_drift_threshold.max_fix_loops_before_escalation`).

---

## S4 — State-specific output

### DONE

1. Write `ratchet-log/iteration-NNN/final-state.json` with verdict.
2. Tell the user: "Iteration N complete. /meta-judge: PASS. gaming_risk_score = X (under threshold 3). <K> findings deduped, <M> non-blocking, 0 blocking. Acceptance fleet exits."
3. End the skill.

### FIX

1. Generate fix-prompt from blocking findings per `references/fix-prompt-template.md`. Cap at 8 bullets.
2. Write `ratchet-log/iteration-NNN/fix-prompt.md`.
3. Tell the user: "Iteration N → FIX. <K> blocking findings. fix-prompt at `ratchet-log/iteration-NNN/fix-prompt.md`. Feed to a fresh implementation session, then re-invoke `/acceptance-fleet` for iteration N+1."
4. End the skill. Do NOT auto-spawn the implementation agent (rule 1 — isolation handoff is the user's call).

### SPEC_DRIFT

1. Write `ratchet-log/iteration-NNN/spec-drift-report.md`. Include: the offending REQs, the recurring counterexamples from /qa-reviewer, the drift signals from /spec-drift-detector, suggested clarify directions.
2. Tell the user: "Iteration N → SPEC_DRIFT. <K> REQs have produced consistent counterexamples across <M> iterations. The spec is likely the bug. Re-invoke `/acceptance-spec <spec_dir>/` with these REQs as the narrow brief. Do NOT continue patching code."
3. End the skill.

### GAMING_RISK

1. Write `ratchet-log/iteration-NNN/gaming-risk-report.md`. Include the gaming vectors that landed (from `/spec-gaming-detector.detected_patterns:`), the `spec_robustness_gaps:` block (concrete contract fixes), suggested `done_when.yaml.behavior.thresholds:` adjustments.
2. Tell the user: "Iteration N → GAMING_RISK. gaming_risk_score = X (threshold 7). Patterns: <list>. Contract has structural gaps the implementer is exploiting. Re-invoke `/acceptance-spec` with this report as the brief — S2.5 will close the surfaced vectors. Do NOT continue patching code."
3. End the skill.

### NEEDS_HUMAN

1. Write `ratchet-log/iteration-NNN/needs-human.md` listing each question verbatim from `/meta-judge.needs_human_items:`.
2. Tell the user: "Iteration N → NEEDS_HUMAN. /meta-judge cannot decide. Required: <list>. Answer in your next message; I'll re-run the classifier with your answer folded in."
3. Do NOT end the skill — wait for user input, then resume at S3.

## S5 — Persist (always runs)

Write the full iteration directory per `references/ratchet-log-format.md`. At minimum:

```
ratchet-log/iteration-NNN/
├── timestamp.txt
├── isolation.json                       # which model went to which sub-skill
├── input-manifest.json                  # checksums of spec / done_when / tests
├── impl-snapshot.tar.gz                 # for next iteration's --history
├── impl-diff.patch                      # vs previous iteration
├── fleet-outputs/                       # raw per-skill outputs
│   ├── code-reviewer-security.yaml
│   ├── code-reviewer-logic.yaml
│   ├── code-reviewer-perf.yaml
│   ├── qa-reviewer.yaml
│   ├── pm-reviewer.yaml
│   ├── spec-drift-detector.yaml
│   └── spec-gaming-detector.yaml
├── meta-judge-output.yaml               # synthesized verdict
├── final-state.json                     # the ratchet state + summary
└── <state-specific report>.md           # fix-prompt | spec-drift-report | gaming-risk-report | needs-human
```

This directory is the contract — anything downstream (debugging, training data, audit) reads from here.

## Models / cross-vendor allocation

The orchestrator itself runs on **Claude Sonnet 4.6** — it's procedural (dispatch + decode), not reasoning-heavy. The actual reasoning lives in the dispatched sub-skills.

If cross-vendor evaluators are available, the orchestrator allocates them to the two slots where cross-vendor matters most: `code-reviewer --adversarial` and `spec-gaming-detector`. See `references/skill-dispatch-matrix.md` for the full allocation logic.

## Tools

- `bash` — for spawning sub-skill invocations
- `read_file` — only for reading sub-skill outputs and bootstrapping
- `parse_yaml` — for ingesting outputs

Critically: no `grep` / `git_log` for the orchestrator's own analysis. The orchestrator does NOT review code. Sub-skills do.

## When to refuse / redirect

- **Inputs incomplete.** Missing `spec.md` / `done_when.yaml` → refuse, suggest `/acceptance-spec` first. Missing `tests/<feature>/` → refuse, suggest `/test-suite-generator` first.
- **Missing `spec-robustness.md`** — do NOT refuse; proceed with a warning that S2.5 was skipped and `/spec-gaming-detector` runs in max-suspicion mode (its own iron rule 10 activates).
- **Cannot guarantee at least medium isolation** — refuse. Single shared session means the impl agent can read the evaluator's prompt next iteration, which destroys the whole anti-gaming structure.
- **`done_when.yaml` has legacy `fitness:` block** — refuse, redirect to `/acceptance-spec` v1.0+ to regenerate per HTML v2 §3.5.
- **No sub-skill is callable** (e.g. `/code-reviewer` not installed in this Claude Code instance) — refuse with the specific missing skill named.

## Cost expectation (per iteration)

7 parallel sub-skill calls + 1 meta-judge synthesis. Rough order of magnitude with prompt caching:

| Sub-skill | Tokens (rough) | Cost (rough) |
|---|---|---|
| code-reviewer × 3 focuses (Opus + Sonnet + Haiku mix) | 50K in / 9K out | $0.20 |
| qa-reviewer (Sonnet + tool use) | 50K in / 10K out | $0.20 |
| pm-reviewer (Opus + tools) | 30K in / 5K out | $0.12 |
| spec-drift-detector (Opus, when triggered) | 25K in / 4K out | $0.10 |
| spec-gaming-detector (Opus, optionally cross-vendor) | 25K in / 4K out | $0.10 |
| meta-judge (Opus only) | 25K in / 4K out | $0.10 |
| Orchestrator overhead (Sonnet) | 5K in / 1K out | $0.02 |
| **Per-iteration total** | | **~$0.84** |

Pipelines converge in 2-4 iterations (the four-state machine catches spec-drift and gaming-risk early instead of looping FIX endlessly). Budget ~$2-4 per feature acceptance.

Cross-vendor evaluators cost slightly more per call but converge faster. Net cost usually lower than same-vendor-only despite higher per-call cost.
