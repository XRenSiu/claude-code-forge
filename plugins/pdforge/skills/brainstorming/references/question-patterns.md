# 高质量提问模式

> 按需加载，提升提问质量

---

## 1. 核心原则

### 每次只问一个问题

```
❌ 差的提问
"你想用什么框架？需要支持移动端吗？性能要求是什么？还有，要考虑国际化吗？"

✅ 好的提问
"我建议用 React，因为团队最熟悉。这个选择可以吗？"
[等待回答]
"好的。接下来，这个功能需要支持移动端吗？根据用户数据，70% 的用户在桌面端使用，我建议先专注桌面端。"
```

### 推荐驱动

```
❌ 推卸判断
"你想要什么优先级？"

✅ 做推荐
"我建议 P2，因为这影响用户体验但不阻塞其他工作流。可以吗？"
```

---

## 2. 提问类型库

### 类型 1：二元确认

用于验证假设或推荐。

**模板**：「我建议 [方案]，因为 [理由]。可以吗？」

```
"我建议使用懒加载，因为列表可能有上千条。可以吗？"
"我理解这个功能只面向付费用户。对吗？"
```

### 类型 2：有限选项

用于需要用户决策的分叉点。

**模板**：「有两个方向：A [描述] 或 B [描述]。我推荐 [X]，因为 [理由]。」

```
"有两个方向：
A) 客户端计算，响应快但消耗设备资源
B) 服务端计算，统一但增加延迟

我推荐 A，因为你们的用户主要用高性能设备。怎么看？"
```

### 类型 3：开放探索

用于需要用户输入新信息的场景。**谨慎使用**。

**模板**：「关于 [主题]，有什么我需要知道的特殊情况吗？」

```
"关于用户权限，有什么我需要知道的特殊情况吗？比如特殊角色或例外规则？"
```

### 类型 4：优先级排序

用于资源有限需要取舍时。

**模板**：「如果只能先做一个，[A] 和 [B] 哪个更重要？我倾向 [X]，因为 [理由]。」

```
"如果只能先做一个，实时同步和离线支持哪个更重要？我倾向实时同步，因为你们的用户大多在线办公。"
```

---

## 3. 提问顺序策略

### 漏斗模型

从宏观到微观，逐步聚焦：

```
1. 目标层：「这个功能要解决什么问题？」
2. 用户层：「主要是什么用户在什么场景使用？」
3. 范围层：「MVP 需要包含什么？可以不包含什么？」
4. 技术层：「有什么技术约束或偏好？」
5. 细节层：「这个边界情况怎么处理？」
```

### 基于侦察的提问

侦察结果决定问题内容：

```
侦察发现：项目用 TypeScript + React + Redux
↓
好问题："我看到你们用 Redux 管理全局状态，新功能也用 Redux 还是考虑局部状态？"
↓
差问题："你们用什么状态管理？"（侦察能回答的问题不要问）
```

## 4. 问题质量检查清单

问每个问题前，检查：

- [ ] 这个问题侦察阶段回答不了吗？
- [ ] 我有推荐吗？如果有，说出来
- [ ] 这是一个问题还是多个伪装成一个？
- [ ] 用户能用一两句话回答吗？
- [ ] 这个问题对设计决策有实质影响吗？

## 5. 常见错误

### 错误 1：问已知信息

```
❌ "你们用什么框架？"
（侦察时已经看到 package.json 了）

✅ "我看到你们用 Next.js 14，新页面要用 App Router 还是保持 Pages Router 一致性？"
```

### 错误 2：问太抽象

```
❌ "你对这个功能有什么期望？"

✅ "这个功能上线后，你期望看到什么指标变化？比如用户停留时间或转化率？"
```

### 错误 3：问不影响决策的细节

```
❌ "按钮用蓝色还是绿色？"
（在概念设计阶段问这种细节）

✅ 等到 UI 实现阶段再问视觉细节
```

### 错误 4：连续追问

```
❌ 
"需要支持移动端吗？"
"那平板呢？"
"响应式还是单独适配？"
"什么断点？"

✅
"我建议先专注桌面端，上线后根据用户数据决定是否适配移动端。可以吗？"
[等待回答]
"明白需要移动端。我建议用响应式设计，断点设在 768px 和 1024px，符合你们现有页面的模式。"
```

## 6. 提问节奏

### 理想节奏

```
问题 → 用户回答（1-2 句）→ Claude 确认理解 → 下一个问题
```

### 避免的节奏

```
问题 → 用户长篇回答 → 问题 → 问题 → 问题（连珠炮）
```

### 提问数量控制

- 简单功能：3-5 个问题
- 中等功能：5-8 个问题  
- 复杂功能：8-12 个问题

**超过 12 个问题说明应该拆分功能。**

## 7. 特殊场景处理

### 场景：用户说「你决定」

```
用户："这个你决定就好"

✅ 做决定并说明理由
"好的，我选择方案 A（服务端渲染），因为这样对 SEO 更友好，符合你们内容站的定位。如果之后需要调整，告诉我。"

❌ 继续追问
"真的吗？你确定？这个决定挺重要的..."
```

### 场景：用户答非所问

```
问题："这个功能面向所有用户还是只对付费用户？"
回答："我们希望尽快上线。"

✅ 重新引导
"明白时间紧迫。关于用户范围，如果暂时只开放给付费用户，可以减少测试范围加快上线。你看这样可以吗？"
```

### 场景：用户不确定

```
回答："我也不太确定，可能需要问产品经理。"

✅ 提供默认方案
"没问题。我先按 [默认方案] 设计，这是比较保守的选择。等你确认后如果需要调整，改动不大。"
```
