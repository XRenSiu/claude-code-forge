# Docs Extraction Prompt Template

This template is for the docs-scanning work: extracting **business vocabulary
signals** from natural-language docs without classifying or judging — that
judgment comes later, when objects are classified and converged.

The output is a structured Markdown file (`01b_docs_terms.md`) that gets fed
into the classification and convergence work alongside the code-side inventory
(`01_inventory.md`).

---

## When to use this template

Use the docs scan when the project has **at least one** of:

- A `README.md` longer than ~50 lines (a real README, not a stub)
- A `docs/` directory with `.md` files
- A `spec/` or `specs/` directory
- A `PRD/`, `prds/`, or `requirements/` directory
- A `design/` or `designs/` directory
- An `ARCHITECTURE.md`, `DESIGN.md`, or similar top-level markdown

If none of these exist, skip the docs scan and classify with the code-only
inventory. Note this in `decisions.md` as a known gap.

---

## The prompt to use

When the docs scan runs, the extraction agent should perform the following work
itself (no separate API call needed — this is what the agent running the skill
should do while scanning docs):

```
Read the following project documentation files: <list of paths>

Your task is to extract BUSINESS VOCABULARY from these documents. You are NOT
classifying, judging, or merging anything — that comes later. You are doing
mechanical extraction of what the team's product language is.

For each candidate term, output:
1. The term itself (preserve the casing the docs use)
2. Approximate frequency across all docs (count actual occurrences)
3. 2-3 short quotes (≤15 words each) showing how the term is used in context
4. The file paths where the quotes came from

Rules of inclusion:
- INCLUDE: nouns that appear to name product entities, user-facing features,
  domain concepts, or recurring abstractions
- INCLUDE: verbs that the docs use to describe what users or the system DO
  (especially when those verbs reappear with the same nouns)
- INCLUDE: terms that the docs explicitly bold/quote/define

Rules of exclusion:
- EXCLUDE: generic words ("user", "feature", "system", "data") UNLESS the docs
  treat them as definitive terms
- EXCLUDE: implementation/architecture words ("API", "database", "service",
  "endpoint") UNLESS they are presented as user-facing concepts
- EXCLUDE: filler words, marketing copy, transitional phrases

Output structure (use this exact Markdown format):

## Nouns

- **<Term>** (≈<count> occurrences)
  - "<quote 1>" — `<file:line if available, else file>`
  - "<quote 2>" — `<file>`
  - Notes: <one optional line if the docs explicitly define or contrast this term>

## Verbs

- **<verb>** (≈<count> occurrences)
  - "<quote showing subject and object>" — `<file>`
  - Notes: <typical subject and object pairings>

## Definitions found

If any docs contain explicit definitions (e.g. "**Topic**: a unit of thought..."),
list them here verbatim. These are gold-tier signals.

## Cross-references

If a doc explicitly contrasts two terms (e.g. "we use 'Topic' not 'Node' because..."),
list these. They are direct evidence for Judgment 2 (merging synonyms).

## Bounded Context hints

If the docs mention multiple subsystems, modules, or "areas" with their own
vocabulary (e.g. "the Identity service handles..."; "in the Billing module..."),
list these. They feed Judgment 4 (splitting contexts).
```

## How to actually run this in practice

The agent executing the skill should:

1. **Discover doc paths.** Look in this order, take what exists:
   - Root: `README.md`, `ARCHITECTURE.md`, `DESIGN.md`, `CONTRIBUTING.md`
   - `docs/`, `doc/`, `documentation/`
   - `spec/`, `specs/`, `specifications/`
   - `PRD/`, `prd/`, `prds/`, `requirements/`
   - `design/`, `designs/`
   - Any other `*.md` at root level

2. **Filter out noise.** Skip:
   - `CHANGELOG.md`, `LICENSE.md`, `CODE_OF_CONDUCT.md`, `SECURITY.md`
   - Auto-generated docs (often have headers like "Generated by ...")
   - API reference dumps that are pure schema repetition

3. **Read in chunks if needed.** If total docs exceed ~50KB, prioritize:
   - README first (sets product framing)
   - Anything in `spec/` or `PRD/` (most likely to have explicit definitions)
   - Architecture docs last (often more about implementation than vocabulary)

4. **Apply the prompt above** to the collected content. Output goes to
   `<workspace>/01b_docs_terms.md`.

5. **Sanity check the output**:
   - If fewer than 5 nouns came out of substantial docs (>20KB), the docs are
     either very abstract or the extraction missed something — re-run with a
     prompt explicitly asking "what are the most-mentioned product nouns".
   - If 50+ nouns came out, the extraction was too liberal — the noise will
     swamp classification. Re-run with stricter exclusion (especially of generic words).

## What good output looks like

Example excerpt for a hypothetical mind-mapping product's docs:

```markdown
## Nouns

- **Topic** (≈47 occurrences)
  - "Each Topic represents a unit of thought" — `docs/concepts.md`
  - "Topics can be connected to form a mind map" — `README.md`
  - "Drag a Topic onto another Topic to nest them" — `docs/canvas.md`
  - Notes: Always capitalized in docs; explicitly contrasted with "node" in concepts.md

- **Sheet** (≈18 occurrences)
  - "A Sheet is a canvas containing one or more Topics" — `docs/concepts.md`
  - "Each Sheet has a single root Topic" — `docs/concepts.md`
  - Notes: Distinct from "tab" or "page"; carries semantic weight

## Definitions found

> **Topic**: A node in the user's thinking graph. Topics have content, position,
> and optional style. — `docs/concepts.md`

> **Sheet**: A bounded canvas containing a connected set of Topics rooted at a
> single root Topic. — `docs/concepts.md`

## Cross-references

> "We deliberately use 'Topic' rather than 'Node' to emphasize the semantic
> weight — Topics are about thought, Nodes are about graph structure." —
> `docs/decisions/0007-topic-vocabulary.md`

This last cross-reference is gold-tier evidence for the Topic-vs-Node merge
decision (Judgment 2).
```

## What to do with this output downstream

This output is an input to three later concerns. The dependency is real — you
can't apply these until the docs scan has produced `01b_docs_terms.md` — but
they're described here as uses of the output, not numbered steps.

### When classifying terms

When deciding whether a term is `business`, `ui`, `impl`, or `unclear`:
- A term that appears in `01b_docs_terms.md` AND `01_inventory.md` → strong
  business signal. Default to `business` unless Judgment 1 catches it as UI/impl.
- A term in `01_inventory.md` but NOT in docs → might still be business, but
  raise the bar. Could be over-engineered code or a private impl detail.
- A term in docs but NOT in code → flag this loudly. Either the concept is not
  yet implemented, or it's implemented under a different name (a Judgment 2
  signal, resolved during convergence).

### When converging objects

The "sentence test" in Judgment 2 becomes much sharper with docs evidence:
- If docs use **`Topic`** consistently and code uses **`Node`** internally,
  Topic wins as canonical — the team's actual vocabulary is in docs.
- The "Cross-references" section of `01b_docs_terms.md` is a direct input:
  any explicit "we use X not Y" statement is a settled merge decision.
- The "Bounded Context hints" feed Judgment 4: if docs separate "Identity"
  from "Workspace" with different vocabularies, that's evidence for splitting.

### When drafting `dos.yaml`

- `scope.in_scope` and `success_criteria`: extract directly from docs.
- `rules` candidates from docs: any sentence containing "must", "always",
  "never", "required" is a candidate constitutional rule. Apply Judgment 3
  to filter.
- `agent_guidelines`: team conventions in CONTRIBUTING.md or design docs
  often translate directly.
- `bounded_contexts`: architecture docs are the primary source.
