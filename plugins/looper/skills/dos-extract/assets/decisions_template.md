**Worked example** (this is what the user explicitly asked about):

> ### Merge: `Topic` ← { `Topic`, `Node`, `TopicNode`, `Item` }
>
> **Judgment 2 application**:
> - Lifecycle test: identical (all are mutable canvas elements with same archive
>   semantics; all are persisted as part of a Sheet).
> - Permission test: identical (Sheet owner controls all of them).
> - Sentence test:
>   - "Sheet CONTAINS Topics" — natural ✓
>   - "Sheet CONTAINS Nodes" — sounds like a graph database concern, not a
>     mind-mapping product
>   - "Sheet CONTAINS TopicNodes" — clunky compound; "Node" is redundant once
>     "Topic" is established
>   - "Sheet CONTAINS Items" — too vague; `Item` could be anything
>
> **Chosen name**: `Topic`
>
> **Rejected names with reasoning**:
> - `Node` — leaks graph implementation. The fact that Topics happen to be
>   stored as nodes in an internal graph structure is an implementation choice;
>   tomorrow we might rearrange them as a tree, a timeline, or a flat list, and
>   "Node" would become a lie.
> - `TopicNode` — same problem as `Node`, with an extra word that doesn't add
>   meaning. Worse, it suggests `Topic` and `Node` are different things, which
>   they aren't.
> - `Item` — semantically empty. The term carries no commitment to what kind
>   of thing this is. The DOS is supposed to constrain the team's vocabulary;
>   `Item` constrains nothing.

## Caveats for the human reviewer

Things to verify before treating `dos.yaml` as ground truth:

1. **Scope inference is fragile.** The `in_scope` and `out_of_scope` fields were
   inferred from documentation. Confirm with the product team.
2. **`audience` field is templated.** Verify against actual stakeholder list.
3. **`success_criteria` likely needs human input.** Hard to extract mechanically.
4. **`open_questions` may be sparse.** Real systems have ambiguities; if this
   section has fewer than 2 entries, the extraction may have papered over them.
5. **`anti_patterns` has only what the code revealed.** Team's historical
   ontology mistakes (pre-this-codebase) won't be captured — add them manually.
6. **Bounded Context detection is conservative.** The skill only splits when
   evidence is strong; subtler splits may have been missed.
