# 溯因恢复 —— 通道二的完整程序(护城河段)

> 溯因通道的细则。这是 invariant-extract 区别于"业界只覆盖演绎半边"的那条通道,也是记忆复利
> 真正长出来的地方。读它,然后才做溯因。

---

## 一、溯因 ≠ 演绎

- **演绎**(通道一):代码里**已经写着**这条检查,你把它显化。信息是 present 的。
- **溯因**(通道二):你**观测到一次失败**,反推"哪条规则被违反,才解释得了它"。信息是
  **缺失**的——正因为那条规则当时没被写下,失败才发生。

> failure.memory 是这把刀的**主输入**:一次失败 = "这条不变量存在、且当时没被写下"的经验证据。

---

## 二、程序(逐 obstacle)

对 failure.memory 里的每一个 obstacle(算子链:**投影 → 取反 → 收窄**):

0. **先按目的投影(aspect 轴)。** 把该 obstacle 投到"这块领地负责的属性"子空间(目的 = name +
   kpi + dos.scope,见 SKILL.md "目的:贯穿全程的取景框")。**投影为 ∅(与本领地目的无关)→ 跳过该
   obstacle,不硬挤规则。** 非排他:同一 obstacle 可被别的领地在它自己的 run 里接走。
   例:纯时延超时,对"正确性领地"投影为 ∅;对"时延领地"非 ∅。
1. **重建它怎么坏的。** 触发条件是什么?当时的状态?错误链 / Verdict 的 reasons?
   产出一句"失败的因果":`在 <条件> 下,因为 <缺失的保证>,导致 <坏结果>`。

2. **取反成候选规则。** 把"缺失的保证"写成一条恒成立式:
   `只要 <X> 一直成立,这次就不会发生`。X 即候选不变量。

3. **取最窄规则(规则上的奥卡姆)。** 一个 obstacle 的取反是**欠定的**——很多条规则都能阻止
   那次失败。**取能覆盖该 obstacle 的最窄那条**,见第三节。

4. **绑死到那个具体 obstacle。** 候选必须携带它来自哪次失败(MemoryAsset id / Run / Verdict 引用)。
   以后可审、可放宽、可在落卡阶段当回归守卫。

5. **标置信度。** 窄而直接的 → 高;宽、跨多系统、或需要假设链才成立的 → 低。**低置信 / 高风险的
   只提案(NEEDS_HUMAN),不自动入卡**(见 anti_patterns 失败模式②)。

每条候选含 `{候选规则, 来源 obstacle, 投影 aspect, 置信度, 最窄性理由}`,落进卡的
hard_invariants / overridable_defaults 候选栏(`assets/invariant_card.yaml`)。

---

## 二点五、两个正交的轴:目的挑维度,奥卡姆挑宽窄

一次失败不是一条规则的取反,而是**一整族规则**,沿两个**正交**的轴散开:

- **aspect(维度)轴** —— 由**目的**挑:正确性 / 时延 / 成本 / 安全 / 可逆性 ……
- **scope(宽窄)轴** —— 由**奥卡姆**挑(下一节):覆盖该 obstacle 的最窄规则。

**奥卡姆只管宽窄;维度由目的定死。沿错误维度取的最窄规则,照样是错的。** 这就是为什么 step 0 的
投影必须在取反之前:它先把 obstacle 钉到正确的 aspect 子空间,奥卡姆才在这个子空间里收窄。

铁证(同代码、同失败、两条不同 □):同一段退款超时——
- **退款正确性领地**(目的=金额正确、不可逆动作不出错)→ □ = "永不在未核验金额下发起退款";
  超时本身投影可能为 ∅(除非它导致了重复退款)。
- **客服时延领地**(目的=响应 SLA)→ 同一超时 → □ = "P95 < Xs,超时须降级而非阻塞"。

aspect 轴有现成载体:**`Territory.kpi` 就是它**——KPI=时延的领地把失败投到时延维度,
KPI=退款错误率的投到正确性维度。不必新造字段。

---

## 三、最窄规则(防过度泛化)

这是溯因最容易做错的一步(失败模式①)。例子:

> 一次**超时**失败。

- 过度泛化:`THE SYSTEM SHALL NEVER call API X`(永不调该 API)——杀过了头,把正常功能也禁了。
- 过度具体:`SHALL NOT call API X at 14:03 on Tuesday`——绑死到无关细节,下次换个时间又坏。
- **最窄且对**:`THE SYSTEM SHALL NEVER call API X without a retry budget`
  ("永不在没有重试预算下调该 API")——恰好覆盖这个 obstacle,不多禁一分。

判最窄的操作:**逐步加回约束,直到规则刚好不再阻止"本该允许的执行"为止**。
- 太宽 → 它会误杀某些合法 Run(想象一个反例 Run);收窄。
- 太窄 → 它放过了同一类的另一个失败变体;放宽到覆盖该类。
- 停在:覆盖该 obstacle 及其同类、但不误杀合法 Run 的那个边界。

最窄规则的好处:**可审、可放宽**。绑死到具体 obstacle 后,将来若发现它仍太宽,有明确的放宽路径
(回到那次失败,重判最窄边界),而不是面对一条来路不明的全局禁令。

---

## 四、与演绎通道的分工(不要混)

- 通道二**不取代**通道一。通道一给"代码已知的"base load;通道二给"代码用代价换来的"那部分。
- 同一条规则若**两条通道都命中**(代码里有 guard + 又出过对应失败):置信度最高,优先落卡,
  且在 decisions.md 记双来源。
- 通道二产出的候选,同样要过 存活测试(□/◊)与强度分类——**溯因出来的不
  自动是 □**,它也可能是一条只对某任务成立的 ◊(那就踢回 done_when 生成器)。
